# 차등성과 집계 방식 — 쉬운 설명서

> 개발이나 영어를 잘 모르시는 분도 이해할 수 있게 풀어 쓴 설명입니다.
> 작성 기준: 2026-05-19

---

## 1. 차등성과가 뭐예요?

매 회차마다 코디들의 영업 성과를 비교해서 **A / B / C / D / H** 다섯 등급으로 자동 분류하는 시스템입니다.

* **A 등급**: 그 회차에서 성과가 가장 좋은 코디들 (상위 1/3)
* **B 등급**: 중간 1/3
* **C 등급**: 하위 1/3
* **D 등급**: 취소를 반영한 계약이 0 이하인 코디 (성과 0 이하)
* **H 등급**: D 등급을 4번 연속으로 받은 코디 (위험 등급)

여기서 핵심은 **취소반영 계약률**(취소된 건수를 뺀 순 계약률)로 줄을 세운다는 점입니다.

---

## 2. 어떤 파일이 필요한가요?

총 4가지 파일을 업로드해야 차등성과가 산정됩니다.

| 파일 | 어디서 받나요 | 어떤 내용인가요 |
|---|---|---|
| **DB수 파일** | WM 고객관리 시스템 | 회차 기간 동안 코디에게 분배된 고객 목록 |
| **계약수 파일** | WM 고객관리 시스템 | 최근 3개월 누적, 계약 상태별 고객 (구매완료, 환불완료 등) |
| **취소수 파일** | 신청자관리 시스템 | 최근 3개월 누적, 반품/환불 발생 신청자 목록 |
| **명단 파일** | 근태보고 시스템 | 코디 명단 (소속·등급 정보) |

### 중요 운영 룰

* **DB수**: 그 회차 분배일 데이터만 (예: 4/11~4/13 분배분만)
* **계약수 / 취소수**: 같은 CMS에서 **오더일 기준 최근 3개월**로 export해서 매번 같은 윈도우로 올리기
* **명단**: 최신 1개만 유지

> 두 파일(계약수·취소수)을 항상 같은 윈도우로 받는 게 중요해요. 그래야 매칭 오류가 줄어듭니다.

---

## 3. "회차"가 뭐예요?

회차는 **DB가 코디들에게 분배된 며칠 사이클**입니다. 예를 들어:

* 회차 A: 5월 4일~5월 8일 (5일간) 분배된 DB
* 회차 B: 5월 11일~5월 13일 (3일간) 분배된 DB
* 회차 C: 5월 14일~5월 18일 (5일간) 분배된 DB

DB수 파일의 **분배일 최소값~최대값**으로 회차가 자동으로 정의됩니다.

회차 이름은 시트명으로 `YYMMDD` 형태(분배일 마지막 날). 예: `260518` = 2026년 5월 18일이 마지막 날.

## 4. 핵심 지표 — 어떻게 셈하나요?

### ① DB수 — 받은 분배 건수

코디가 회차 기간 동안 분배받은 DB 개수입니다.

**카운트 조건** (모두 만족하는 행만 +1):

| 조건 | 설명 |
|---|---|
| 분배일이 회차 안 | DB 파일의 분배일 칼럼이 회차 min~max 범위 |
| 분배일이 평일 | 주말·공휴일에 분배된 건 제외 |
| 멤버 코디 | 명단의 W/E/A/B/H 그룹 중 뇌새김영어 소속 |
| `재콜` 표시 없음 | DB정보 필드에 '재콜' 단어 포함 시 제외 |
| 매체에 `talkis` 없음 | 톡이즈 분리 운영 |
| `관리자/군자/삭제요청` 코디 제외 | 비코디 계정 |
| `IN` 무매체 제외 | DB정보에 'in' 포함하고 매체 공란 = 제외 (단 2026-05-18 이후 업로드 + 정확히 'IN' 글자면 예외적으로 포함) |

### ② 실계약수 — 계약 + 정상취소를 합친 수

코디가 회차에 분배받은 고객 중 **계약을 받아낸 수**입니다. 계약 후 정상 반품된 것도 포함됩니다(코디 입장에선 계약을 한 번 받아낸 거니까).

**카운트 조건**:

| 조건 | 설명 |
|---|---|
| 오더일이 회차 안 | 계약수 파일의 오더일이 (직전 회차 분배 마지막날, 현재 회차 분배 마지막날] 범위 — 수학 표기 (a, b]는 "a 초과 b 이하" |
| 분배일이 평일 | 휴일 분배 제외 (오더일은 평일·휴일 무관) |
| 멤버 코디 | 명단 매칭 |
| 매체·관리자 필터 | talkis 매체, 군자·관리자·삭제요청 코디 제외 (DB수와 동일) |
| IN 무매체 룰 | DB수와 동일 (2026-05-18 이후 'IN' 정확매칭은 포함) |
| 재콜 | **포함** (DB수에서는 제외하지만 실계약·취소수에서는 포함) |
| 고객상태 = 계약 또는 정상취소 | (아래 §5 참조) |

### 회차 매칭 — `(a, b]` 표기 자세히

* 예: 직전 회차 = 260513 (분배 5/11~5/13), 현재 회차 = 260518 (분배 5/14~5/18)
* 현재 회차 오더일 범위 = **(5/13, 5/18]** = 5/14 ~ 5/18
* 5/13에 오더된 행 → 직전 회차에 잡힘 (이미 처리됨)
* 5/14 ~ 5/18에 오더된 행 → 현재 회차 실계약 +1
* 5/19 이후 오더된 행 → 다음 회차에 잡힘

회차들 사이 gap(예: 5/14 분배 직전 5/13 23:59 오더)도 직전 회차 max 이하라 자동으로 직전 회차에 흡수됩니다. **같은 오더가 두 회차에 잡히지 않고 정확히 한 곳에만 매칭**됩니다.

**예시**:
```
분배일 5/12(현재 회차) + 오더일 5/15(현재 회차) + 상태 배송중 → 실계약 +1 ✅
분배일 5/12 + 오더일 5/15 + 상태 반품입고 + 취소수 파일에 매칭 O → 실계약 +1 (정상 취소)
분배일 5/12 + 오더일 5/15 + 상태 반품입고 + 취소수 파일에 매칭 X → 실계약 ❌ (배송전 취소로 보고 제외)
분배일 3/15(과거 회차) + 오더일 5/12(현재 회차) + 배송중 → 실계약 +1 ✅ (분배가 과거여도 OK)
```

> 💡 **포인트**: "내가 분배받은 DB가 결국 얼마나 계약으로 이어졌나"가 아니라 **"이번 회차에 오더가 발생한 계약이 몇 건인가"**로 셉니다.

### ③ 취소수 — 두 갈래로 구성

**(1) 정상 반품** — 취소수 파일에서 반품일이 회차 안인 행

| 조건 | 설명 |
|---|---|
| 반품일 회차 안 | 취소수 파일의 반품일(AQ열)이 회차 분배일 min~max 범위 |
| 분배일 평일 | 휴일 분배 제외 |
| 코드에 'in' 없음 | R열 코드번호 |
| 담당자가 멤버 | N열 매니저 매칭 |
| 재콜 | **포함** (취소수 파일은 신청자관리 형식이라 재콜 정보 자체가 없음) |

**(2) 사후 배송전 취소** — 직전 회차 취소수 파일에는 있었는데 이번 회차 파일에는 없는 고객

배송전 취소가 발생하면 신청자관리에서 그 행이 삭제됩니다. 그래서 두 파일을 비교해 사라진 고객키를 사후 배송전 취소로 간주합니다.

| 조건 | 설명 |
|---|---|
| 직전 cancel D열에 있었음 | 어제 사용한 취소수 파일의 고객키 |
| 이번 cancel D열에는 없음 | 오늘 올린 취소수 파일에서 사라짐 |
| 자연 cutoff drop 아님 | 직전 행 분배일이 이번 cancel 시점 - 3개월 이후 (오더일 3개월 외 자연 만료는 제외) |
| 담당자 멤버 매칭 | 그 코디 cancel +1 |

> 💡 **이 분기는 다음 회차부터 작동합니다.** 첫 회차에는 비교 대상(직전 cancel)이 없어서 못 잡아요.

### ④ 취소반영 계약수 — 등급 결정의 핵심 지표

```
취소반영 계약수 = 실계약수 - 취소수
```

예: 실계약 10건 - 취소 3건 = 취소반영 계약수 7건

**취소반영 계약률** = 취소반영 계약수 / DB수

이 값(%)이 등급 결정의 1차 기준입니다.

## 5. 고객상태 분류 — 계약/취소/제외/기타

계약수 파일(WM 고객관리) AD열의 "고객상태" 값에 따라 분류:

| 분류 | 키워드 | 의미 |
|---|---|---|
| **계약** | 배송대기, 배송중, 자동구매완료, 구매완료, AS | 정상 계약 |
| **취소 후보** | 반품입고, 승인요청, 입고대기, 환불요청, 환불완료 | 취소 발생 (단 cancel 파일과 매칭 여부에 따라 정상 취소/배송전 취소 갈림) |
| **제외** | 결제보류, 상담취소, 상담요청, 상담대기, 결제요청, 결제변경후 요청, 신용정보 보류 | 계약·취소 어디에도 안 들어감 |
| **기타** | 위에 해당 안 되는 값 | 무시 |

### 배송전 취소 판별 핵심

* 상태가 **반품입고/환불완료** 등 취소 후보일 때:
  - 취소수 파일(신청자관리) D열 고객키에 **매칭 O** → 정상 반품 → 실계약수 포함
  - 취소수 파일에 **매칭 X** → 배송전 취소 → 실계약수에서도 제외, 취소수에도 안 들어감 (현재 회차 신규 배송전 취소는 양쪽 다 제외)

## 6. 휴일 처리 — 분배일 기준이 원칙

* **휴일 판정 기준은 항상 "분배일"**
* DB수, 실계약수, 취소수 카운트 모두 **분배일이 주말·공휴일이면 제외**
* 오더일이 휴일인지는 보지 않음 (그래서 평일 분배 + 휴일 오더는 카운트됨)

**왜 그런가요?** 회차 자체가 분배일 사이클로 정의되므로 분배일이 회차에 안 들어오면 그 행은 회차의 대상이 아닙니다.

## 7. 등급 자동 산정 룰

### 1단계: 부정 분류 — D 등급

* **취소반영 계약수가 0 이하**인 코디 → D 등급 자동
  - 예: 실계약 3건 - 취소 5건 = -2 → D 등급
  - 분배 자체를 못 받은 코디(DB수 0)도 자연스럽게 D로 분류

### 2단계: 양수 분류 — A/B/C

남은 인원(취소반영 계약수 > 0)을 **취소반영 계약률 내림차순**으로 정렬 후 1/3씩 분할:

* 상위 1/3 → A
* 중간 1/3 → B
* 하위 1/3 → C

**동률 처리**: 1/3 경계에서 같은 % 인원은 위쪽 그룹에 묶입니다.

### 3단계: H 등급 — 위험 누적

* D 등급을 **4회 연속 받으면 H 등급으로 강등**
* H 등급에서 벗어나려면: 직전 회차 H + 이번 회차 계약 2건 이상 → C 등급으로 복귀

### 4단계: 신입 보호

* 입사 후 **60일 이내 신입**은 자동으로 D/H 받으면 → **C로 상향**
* 이미 C 이상이면 그대로 유지

### 5단계: 정당사유 회차 유지

* 회차 기간 동안 모든 일자가 "정당사유"로 처리됐고 출근이 1일도 없으면 → **직전 회차 등급 유지**
* 정당사유 회차는 자동 산정 자체에서 제외

### 6단계: 근태 강등

* 회차 내 근태 이슈(결근·지각 등)가 **2건 이상**이면 → 한 단계 강등

### 7단계: 1일 출근 cap

* 3일 회차 중 1일만 출근한 경우 → **B 상한** (A 절대 불가)

## 8. 프로모션 1등 — 한 단계 위로 승급

각 회차의 자동 산정 후 **B의 contract 1등, C의 contract 1등**을 한 단계 위로 승급시키는 별도 룰입니다.

### 작동 방식

| 조건 | 결과 |
|---|---|
| B 그룹 contract 1등 → A로 승급 | 비고에 "실계약수 프로모션 1등" 표시 |
| C 그룹 contract 1등 → B로 승급 | 비고에 "실계약수 프로모션 1등" 표시 |

### 풀에서 제외되는 경우

다음 코디는 promotion 후보에서 빠집니다:

* 정당사유 회차 (직전 등급 유지)
* H 등급에서 막 탈출한 사람
* 신입 보호자 (60일 이내)
* DB 0건
* 근태 이슈 2건 이상
* B→A 승급의 경우 출근 1일만 한 사람

### 수동 우선순위

사용자가 [수정] 모드에서 어떤 코디를 그 그룹으로 옮겼으면 → **그 사람들 중에서 contract 1등** 선택. 자동 멤버보다 우선합니다.

> 💡 박연정을 수동으로 B로 옮기면 → 박연정이 B의 수동 멤버 → 자동 멤버(예: 정경화)보다 우선해서 1등으로 picked → A로 승급.

### 동률 시 동시 승급

1등의 모든 표시 데이터(DB수, 실계약, 실계약률, 취소수, 취소반영, 취소반영률, streak)가 완벽히 같은 인원은 모두 함께 승급됩니다.

## 9. 수동 조정 기능

화면에서 직접 수정 가능한 항목들. 모든 조정은 **해당 회차에만 적용**되고 다음 회차에 영향 없습니다.

### ① 등급 수동 변경

* 차등성과 화면 → [수정] 클릭
* 코디 등급 드롭다운으로 변경 (A↔B↔C↔D↔H)
* [적용]
* → manual 플래그가 박혀서 자동 산정과 무관하게 그 등급 고정

자동 산정으로 복원하려면:
```javascript
await gdClearManual('(R_대전)코디이름');
```

### ② 실계약수 ± 조정

* [수정] 모드에서 실계약수 셀의 `+` / `-` 버튼 클릭
* 1씩 증감
* 회차 산정 시 자동으로 누적 적용

### ③ 취소수 ± 조정 (2026-05-19 추가)

* 실계약수와 동일하게 `+` / `-` 버튼
* 사후 배송전 취소가 잘못 잡혔거나 누락된 경우 보정용

### ④ D등급 연속 횟수 ± 조정

* H 진입 판정 직접 조정

조정값은 회차 옆에 색상 뱃지로 표시되고, 뱃지 클릭으로 초기화 가능합니다.

## 11. 자주 묻는 질문

### Q1. 어제 결과가 오늘 보면 다른데요?
**A.** 코드 업데이트나 새 데이터 업로드로 자동 재계산될 수 있습니다. 어제 값을 그대로 보존하고 싶으면 수기로 수동 조정(±)을 해서 그 회차에 박아두세요. 다음 회차에는 영향 없습니다.

### Q2. 사후 배송전 취소가 한 회차에 너무 많이 잡혀요
**A.** 직전 회차 취소수 파일과 비교해서 사라진 키를 모두 잡습니다. 너무 많이 잡히면:
* 정경화 등 특정 코디 manual 해제하면 그 사람 자동 재산정
* 또는 [수정] 모드에서 취소수 −1 수동 조정으로 직접 보정

### Q3. promotion 1등이 내가 원하는 사람이 아니에요
**A.** [수정] 모드에서 원하는 사람을 그 사람이 와야 할 그룹으로 옮기세요. 시스템이 "수동으로 이 그룹에 옮긴 사람 = 1등 후보"로 해석합니다.

### Q4. 누가 manual 상태인지 어떻게 보나요?
**A.** [수정] 모드 진입 후 등급 셀에 드롭다운이 보이는데, 자동/수동 표시가 행마다 다릅니다. 또는 콘솔에서:
```javascript
gdResult.savedRows.filter(r => r.manualGroup).map(r => r.name);
```

### Q5. cancel 파일은 매번 같은 윈도우(오더일 3개월)로 받아야 한다는데, 다른 윈도우로 받으면요?
**A.** 사후 배송전 취소 판정에서 false positive가 생길 수 있습니다. cont/cancel 두 파일을 항상 같은 cutoff로 운영하세요.
