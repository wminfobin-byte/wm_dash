# N그룹 차등 등급 시스템 — 전체 판정 로직

> 본 문서는 **wm_dash N그룹 성과 탭**에서 코디별 등급을 결정하는 전체 판정 로직을 단계별로 정리한 것입니다.
> 실제 구현 기준: `index.html`의 `generate()`(L.1992~) + `assignGroupsSmart()`(L.2360~).

---

## 0. 한눈에 보기

### 등급 체계
| 등급 | 의미 | 표시색 |
|---|---|---|
| **N_A그룹** | 상위 — 고성과 | 🟦 파랑 (#4361EE) |
| **N_B그룹** | 중위 — 표준 | 🟩 청록 (#2EC4B6) |
| **N_C그룹** | 하위 — 저성과 | 🟧 주황 (#F0920E) |
| **H그룹** | 페널티 — 저성과 장기 지속 | 🟥 빨강 (#D1182B) |

### 판정의 단일 기준 지표
```
취소율반영계약률  =  실계약수 × (1 − 기준취소율)  /  DB수
                                      ↑
                              min(개인 취소율, 40%)
```
**이 값이 높을수록 상위 그룹으로 배정된다.**

### 판정 순서 (6단계 — 순서 엄격)
```
① 기본 3분할 배정    (성과 순위 기반)
② H그룹 복귀 적용    (패자부활)
③ 신규 보호 적용      (입사자 하한선)
④ 수동 수정 복원      (관리자 결재)
⑤ Streak 계산        (연속 카운트)
⑥ H그룹 강등 적용    (만성 저성과 제재)
```
> 각 단계는 앞 단계 결과 위에 덮어쓰여진다. 순서가 바뀌면 결과가 달라진다.

## 2. 기준 지표의 계산 (전제)

```
DB수           = countDb(...)              # 주말·공휴일·재콜·talkis 등 제외
실계약수        = countContracts(...)       # '계약' + '취소' 모두 포함
계약률          = 실계약수 / DB수

기준취소율     = min(개인 분기 취소율, 0.40)   # 40% 상한
취소율반영계약수 = 실계약수 × (1 − 기준취소율)
취소율반영계약률 = 취소율반영계약수 / DB수    ← 판정 기준
```

> 40% 상한이 없으면 신입 한두 건 취소로 100%가 되는 극단치가 하위 그룹을 교란한다.

## 3. ① 기본 3분할 배정 (`assignGroupsSmart`)

### 3-1. 알고리즘
1. 전원을 `취소율반영계약률` 내림차순으로 정렬
2. 기본 커트라인: `cut1 = ⌈n/3⌉`, `cut2 = ⌈2n/3⌉`
3. 상위 `cut1`명 → **N_A**, 그 다음 `cut2 − cut1`명 → **N_B**, 나머지 → **N_C**

### 3-2. 동점자 보정 (같은 성과는 같은 그룹)
소수점 둘째자리까지 반올림한 값이 같으면 동률로 본다.
```javascript
while(cut1<n && roundRate(entries[cut1].rate)===roundRate(entries[cut1-1].rate)) cut1++;
while(cut2<n && roundRate(entries[cut2].rate)===roundRate(entries[cut2-1].rate)) cut2++;
if(cut2<=cut1) cut2=cut1+1;   // 경계 역전 방지
if(cut2>n) cut2=n;
```
- 경계 바로 밑 사람과 값이 같으면 커트라인을 **아래로 한 칸 밀어낸다** (동률자를 위 그룹에 포함시키는 방향).
- 결과: 같은 성과를 가진 두 명이 서로 다른 그룹으로 갈라지는 일이 없다.

### 3-3. 인원 부족 시 (n ≤ 3)
- 1명: N_A
- 2명: N_A, N_B
- 3명: N_A, N_B, N_C

### 3-4. 출력
- `groups`: `{이름: 'N_A그룹' | 'N_B그룹' | 'N_C그룹'}`
- `cutlines`: `{aToBVal, bToCVal, aToBIdx, bToCIdx}` — 커트라인 값(그룹 요약 카드 표시용)

## 4. ② H그룹 복귀 — 패자부활전

### 조건
```
직전 기간의 등급이 H그룹
    AND
이번 기간 실계약수 ≥ 2
    AND
직전 기간 이력 타임스탬프가 HISTORY_START (2026-04-02) 이후
```

### 처리
→ `groups[name] = 'N_C그룹'` (강제로 N_C로 올림. §3의 자동 배정 결과 덮어쓰기)

### 왜 N_A/N_B가 아니라 N_C인가
- H그룹은 DB 배분 자체가 적을 수 있어 성과 수치가 비교 대상이 안 됨.
- 일단 N_C로 "복귀"시킨 뒤 다음 기간에 정상적인 3분할 평가를 받게 한다.
- §6의 Streak 카운트에서 직전 H는 연속 카운트에 포함되므로, 복귀 후 바로 또 N_C가 되면 누적되어 재차 H로 갈 수 있다.

### 데이터 소스
- 직전 등급: `localStorage.ngroup_history[name][0]`
- 타임스탬프: `localStorage.ngroup_history_times[name][0]` (예전 기록 제외용 안전장치)
- 이번 실계약수: `stats[name].contract`

## 5. ③ 신규 입사자 보호

### 조건
```
memberHireDates[name]가 존재
    AND
현재 날짜 ≤ (입사월 + 1개월)의 말일 23:59:59
    AND
§4까지의 결과가 N_C 또는 H
```

### 처리
→ `groups[name] = 'N_B그룹'`
→ `protectedMembers.add(name)` (정렬 시 N_B 최하단에 배치하기 위한 플래그)

### 보호 기간 계산 (예시)
```
입사일        보호 만료
2026-03-15 → 2026-04-30 23:59:59
2026-04-01 → 2026-05-31 23:59:59
2026-04-30 → 2026-05-31 23:59:59
```
`new Date(year, month+2, 0, 23, 59, 59)` — 다음 달의 0일 = 해당 월의 말일.

### 주의
- N_A나 N_B로 자동 배정되었다면 보호가 **적용되지 않는다** (원래 기준보다 더 내리지는 않음).
- 보호 대상은 정렬에서 "N_B 블록의 맨 아래"로 배치되어 실력으로 N_B 안에 든 사람과 시각적으로 구분된다.
- H그룹으로 간 사람도 보호 대상이면 N_B로 상향. 즉 "H그룹인 신입"은 존재하지 않는다.

## 6. ④ 수동 수정 복원

### 조건
```
IDB `results` store에 현재 sheetName(=DB 분배일 마지막 날 YYMMDD)의 기록이 있고
    AND
그 기록의 row에 manualGroup:true 플래그가 있으며
    AND
해당 이름이 이번 멤버 목록에 포함
```

### 처리
→ `groups[name] = savedRow.group` (사용자가 그룹 수정 UI로 저장한 값 복원)

### 의의
- 관리자가 "이 건은 사정이 있으니 올린다/내린다"고 결재한 기록을 재계산 후에도 유지.
- §5 신규 보호 후에 실행되므로, 관리자의 수동 판정이 신규 보호를 오버라이드할 수 있다.
- `manualGroup` 플래그는 `applyGroupEdit()`에서 설정되며, 해당 이름이 실제로 이번에 변경되지 않았어도 보존된다 (`prevPeriod.rows` 보존 로직, L.2427~).

## 7. ⑤ Streak 계산

### 무엇을 세는가
**"현재 기간을 포함해서 N_C 또는 H그룹에 연속해서 몇 번 있었는지"**
(H그룹 기간도 연속 카운트에 포함 — 그래야 만성 저성과자를 제대로 잡는다)

### 알고리즘
```javascript
if (groups[name] === 'N_C그룹' || groups[name] === 'H그룹') {
  count = 1;  // 현재 기간부터 카운트 시작
  // IDB에 저장된 이전 기간 결과를 최신 → 과거 순으로 스캔
  for (const prev of olderResults) {  // sheetName 내림차순
    const row = prev.rows.find(r => r.name === name);
    if (row && (row.group === 'N_C그룹' || row.group === 'H그룹')) count++;
    else break;  // 연속이 깨지면 중단
  }
} else {
  count = 0;
}
```

### 기준점 (STREAK_START)
- `'260402'` (2026-04-02)
- 이 이전 sheetName의 기록은 스캔에서 제외 → 시스템 도입 이전 기간 배제

### 수동 보정 (`ngroup_streak_adj`)
```
localStorage.ngroup_streak_adj = {"이은정": -1, "김영미": 1, ...}
```
- `adjustStreak(name, delta)` UI 버튼으로 동작 (테이블의 streak 배지 클릭 → -1)
- 산출된 count에 delta를 더해 최종 streak 결정 (0 미만은 0으로 절삭)
- H그룹 복귀자 등 정책적으로 카운트를 리셋해야 할 때 사용

### 출력
- `streaks[name]`: 최종 연속 카운트 (0, 1, 2, 3, 4, 5...)
- 대시보드에서 원형 배지 색으로 표시:
  - 0: 회색, 1: 주황, 2: 진주황, 3+: 빨강

## 8. ⑥ H그룹 강등 — 만성 저성과 제재

### 조건
§7에서 계산된 streak이 **4 이상** AND 현재 그룹이 N_C
```javascript
if (groups[name] === 'N_C그룹' && count >= 4) {
  groups[name] = 'H그룹';
}
```

### 왜 4인가
- 1기간(주간/월간)만으로는 판단 불가. 3기간이면 추세 확인.
- **4기간 연속 N_C/H면 구조적 문제** — 개인별 코칭이나 리어사인 필요.
- "3연속 N_C → H" 로직에서 2026-04 리팩터로 "4연속"으로 조정됨 (과도한 페널티 방지).

### 주의: 이 단계에서만 H그룹으로 신규 진입한다
- §3 기본 배정은 H그룹을 만들지 않는다 (N_A/N_B/N_C만).
- §6 수동 수정에서는 관리자가 H로 지정 가능.
- 자동 로직으로 H가 되는 경로는 이 단계가 유일.

## 9. 판정 흐름 — 단일 코디 추적 예시

### 케이스 A: 신규 입사자 (3월 15일 입사, 4월 16일 평가)
```
stats: db=12, contract=2, refCancelRate=0.08, refAdjRate=0.153
§3 → 성과 최하위 → N_C그룹
§4 → 직전 이력 없음 → 변화 없음
§5 → 4월 말까지 보호 기간 → N_B그룹 (protected)
§6 → 수동 수정 없음
§7 → 현재 N_B이므로 count=0
§8 → N_B이므로 미적용
최종: N_B (신규 보호)
```

### 케이스 B: 3연속 N_C 저성과자
```
과거 IDB: 260409 N_C, 260402 N_C
stats: refAdjRate=0.12 → 하위
§3 → N_C
§4 → 직전 N_C (H 아님) → 변화 없음
§5 → 신규 아님
§6 → 수동 없음
§7 → 현재 N_C 연속, count = 1(현재) + 1(260409) + 1(260402) = 3
§8 → count < 4 → N_C 유지
최종: N_C (streak 3)
```

### 케이스 C: 4연속 N_C → H그룹 강등
```
과거 IDB: 260409 N_C, 260402 N_C, 260326 N_C
§3 → N_C
§7 → count = 1 + 3 = 4
§8 → count ≥ 4 → H그룹
최종: H그룹
```

### 케이스 D: H그룹 복귀자
```
과거 IDB: 260409 H그룹 (이전 강등)
stats: db=6, contract=3 (≥ 2건 달성!)
§3 → 성과 중간 → N_B (가정)
§4 → 직전 H이고 2건 이상 → N_C로 강등(복귀 처리)
§7 → N_C로 count = 1 + 1(260409 H) = 2
§8 → count < 4 → N_C 유지
최종: N_C (복귀)
```

### 케이스 E: 관리자 수동 승격
```
자동 판정: N_C (count=3)
§6 → IDB에 manualGroup:true, group:'N_B그룹' 기록 있음
→ groups[name] = 'N_B그룹'
§7 → 현재 N_B이므로 count=0 (이전 N_C였어도 수동 수정으로 리셋)
§8 → N_B이므로 미적용
최종: N_B (수동)
```

## 10. 경계 케이스 & 트랩

| 상황 | 동작 | 주의 |
|---|---|---|
| DB수=0인 코디 | refAdjRate=0 → 최하위에 몰림 | §3에서 N_C 확정. §7 시작하면 count 누적. |
| 같은 refAdjRate 동점자 | §3 동점자 보정으로 같은 그룹 | 반올림 둘째 자리 기준. 셋째 자리만 다르면 동률로 본다. |
| 입사일 정보 없음 | `memberHireDates[name]`가 undefined | §5 적용 X. 신입이어도 보호 안 받는다 → 명단 파일 입사일 컬럼 확인 |
| 직전 H + 이번 계약 1건 | §4 복귀 미충족 | H 유지 (자동 배정은 N_C로 갔을 수 있지만 이전 H라는 이력 때문에 복귀 로직이 발동 안 함) |
| §4 복귀 후 §5 보호 (신입 + 직전 H) | §5에서 N_B로 한 번 더 상향 가능 | 단 "직전 H"인 신입은 거의 없다 (신입이 H까지 가려면 시간이 필요) |
| §6 수동 수정 후 §8 H 강등 | §8이 마지막이므로 덮어쓸 수 있음 | 수동으로 N_C를 고집했는데 streak ≥ 4면 자동 H로. 반대로 수동 N_B는 유지됨 |
| IDB results 비어 있음 (첫 실행) | §7에서 olderResults = [] | 현재 N_C면 count=1. §8 강등 발생 X. |
| 과거 IDB에 H그룹만 여러 번 | §7 H도 연속에 포함 | 4번 째 기간에서 N_C 됐다면 count가 1~4 어디든 될 수 있음 |

## 11. 정렬 — 판정 후의 화면·엑셀 표시 순서

판정 결과를 화면/엑셀에 표시할 때의 규칙 (판정 자체와 별도):

```
1. H그룹은 무조건 최하단
2. 비-H 멤버를 "일반 멤버"와 "신규 보호 멤버"로 분리
3. 각각 refAdjRate 내림차순 정렬
4. 신규 보호 멤버를 N_B그룹의 마지막 위치에 삽입
   (B그룹이 없으면 N_A그룹 마지막 뒤)
5. 그 뒤 H그룹 (refAdjRate 내림차순)
```

`renderTable()` L.2320~ 과 엑셀 `buildSheet()` L.2524~ 가 **완전히 동일한 로직**을 공유한다.

## 12. 판정 로직 튜닝 포인트

로직 자체를 바꾸지 않고 파라미터만 조정할 수 있는 지점:

| 파라미터 | 현재값 | 의미 | 위치 |
|---|---|---|---|
| 기준취소율 상한 | 0.4 | 극단치 제어 | L.2027 `Math.min(..., 0.4)` |
| H그룹 강등 임계 | 4 | 연속 N_C/H 카운트 | L.2098 `count>=4` |
| H그룹 복귀 계약 임계 | 2 | 직전 H에서 복귀하려면 | L.2043 `>=2` |
| 신규 보호 기간 | 입사월+1개월 말일 | 2개월차 말까지 | L.2056 `hireDate.getMonth()+2, 0` |
| Streak 기준점 | `'260402'` | 이 이전 기록은 무시 | L.2080 `STREAK_START` |
| 이력 기준점 (History) | `2026-04-02` epoch | 예전 localStorage 이력 배제 | L.2405 `HISTORY_START` |
| 동점 판정 정밀도 | 소수 2자리 | `roundRate` 구현 | L.2398 `Math.round(v*10000)/100` |

## 13. 판정이 실패할 수 있는 상황 (운영 주의)

1. **명단 파일에 이름이 누락** → 해당 코디가 집계 자체에서 빠진다 (멤버 목록에 없음)
2. **DB/계약 파일에 OB명이 달라짐** → 집계 0. 다음 기간까지는 자동으로 복구 안 됨 (이름 재표준화 필요)
3. **분배일 누락** → DB수 0으로 집계됨
4. **취소율 산정 파일이 오래됨** → 기준취소율이 현실과 괴리. 분기마다 최신화 필요
5. **수동 수정 중 generate 재실행** → manualGroup 플래그는 sheetName 단위로 저장되므로, 같은 기간 내에서는 복원됨. 다른 기간이면 복원 X.
6. **IDB 데이터 손상** → Streak 계산이 부정확. `indexedDB.deleteDatabase('nGroupDashboard')` 후 전 기간 재업로드 필요
