# 3일 차등 성과 집계 대시보드 — Claude AI 제작용 명세서

> **목적**: 뇌새김영어 **대전·유성 센터** 3일 차등 판정 시스템을 위한 성과 집계 대시보드를 Claude AI가 단일 HTML 파일(JS 인라인)로 제작할 수 있도록, **집계 공식·필터 규칙·등급 배정 로직·UI 구성·파일 영속성**까지 모두 정의한 통합 문서.
>
> **기반**: 기존 `wm_dash/index.html` (N그룹 성과 집계) 의 구현 구조를 계승하되, **3일 차등 새 정책**(A/B/C/H 4등급 · 취소 반영 계약률 최우선 · 4회 연속 C → 5회차 H · 한 회차 실계약 2건 H 탈출 · 정당사유 직전등급 유지 · 근태 하나씩 강등 · 신입 60일 보호 · 대전·유성 통합 풀) 을 덮어쓴다.
>
> **출력 목표**: `grade_dash/index.html` 단일 HTML 파일, JS 인라인, IndexedDB 영속 + 파일 백업/복원, XLSX 내보내기, **모바일 반응형**, 브라우저에서 바로 작동.

## 0. 한눈에 보는 파이프라인

```
[4종 입력 엑셀]                [멤버 필터링]                  [건별 집계]
  attend (명단)        ──►   뇌새김영어 · 대전/유성   ──►   코디별 DB / 실계약 / 취소
  db (DB수)            ──►   복합키: '대전_이은정'
  cont (계약수)        ──►                                   [지표 계산]
  cancel (반품)        ──►                                  계약수 · 취소수
  attendance (근태)    ──►                                  → 취소 반영 계약수/계약률
                                                                   │
                                                                   ▼
                                                            [등급 자동 배정]
                                                            reflected_rate 순위
                                                            → A 25% / B 40% / C 35%
                                                            (대전·유성 통합 풀)
                                                                   │
                                                                   ▼
                                                            [특수 규칙 적용]
                                                                   │
                                                                   ▼
                                                              [최종 등급]
```

### 적용 순서 (★ 매우 중요)

```
①  인원 분류           (경쟁대상 / 확정 H / 퇴사 / 신입)
②  취소수 집계         (고객키 매칭 → 달력상 3개월 계약 × 이번 회차 반품)
③  순위 정렬           (reflected_rate DESC → reflected_contract → raw_rate → contract_count → db_count)
④  비중 배정           (상위 25% = A / 40% = B / 35% = C, 참고값)
⑤  H 탈출 체크        (확정 H가 이번 회차 실계약 2건 ≥ → C, streak 리셋)
⑥  H 진입 판정        (직전 4회차 연속 C + 이번 회차 C → H)
⑦  DB 0건 처리        (DB 0 + 정당사유 X → 랭크 최하단)
⑧  정당사유 미출근    (회차 3일 전체 미출근 → 직전 출근 회차 등급 유지, 이후 단계 skip)
⑨  프로모션            (B 1등 → A / C 1등 → B, H 탈출자·신입 보호자 제외)
⑩  근태 강제 강등     (근태이슈 2회 이상 → 한 단계 강등, 단 C는 C 유지)
⑪  신입 보호          (입사 +60일 이내 C/H → B 상향)
⑫  streak_adj 적용    (관리자 수동 streak 조정)
⑬  수동 오버라이드    (IDB의 manualGrade 플래그로 최종 덮어쓰기)
```

| `db` (DB수) | `WM_고객관리_YYYY-MM-DD.xlsx` | 분배일 기준 DB 집계 | 누적 |
| `cont` (계약수) | `WM_고객관리_YYYY-MM-DD.xlsx` | 오더일 기준 계약 집계 | 누적 |
| `cancel` (반품) | `WM_반품_YYYYMMDD.xlsx` (별도 양식) | 고객키 + 반품일 | 누적 |

- db/cont: **동일 포맷** (WM_고객관리). 목적·기간만 다름.
- cancel: **별도 양식**. 최소한 `고객키`, `반품일` 컬럼 필수.
- attendance: **CMS 2-8 근태 데이터 파일 업로드**. 코디명·일자·근태상태 컬럼.
- 업로드 파일은 **고객키 기준 중복 제거** (같은 고객키가 2번 나오면 최신 1건만 사용).

## 3. 멤버 추출 — 명단 파일

### 3-1. 복합키 ★ 중요

동명이인 처리를 위해 `센터_이름` **복합키**로 관리한다.

```js
function getMembers(attendData) {
  const members = [];               // ["대전_이은정", "유성_김동명", ...]
  const memberHireDates = {};       // { key: Date }
  const memberCenters = {};         // { key: 'daejeon' | 'yuseong' }
  const memberDisplayNames = {};    // { key: "이은정" }

  for (const r of attendData) {
    if (r['담당'] !== '뇌새김영어' || !r['이름']) continue;
    const center = inferCenter(r);  // '대전' | '유성' | null
    if (!center) continue;

    const key = `${center}_${r['이름']}`;  // 복합키
    members.push(key);
    memberCenters[key] = center === '대전' ? 'daejeon' : 'yuseong';
    memberDisplayNames[key] = r['이름'];
    if (r['입사일']) memberHireDates[key] = parseDate(r['입사일']);
  }
  return { members, memberHireDates, memberCenters, memberDisplayNames };
}

function inferCenter(row) {
  // 1순위: 명단 파일에 '센터' 컬럼이 있으면 그 값
  if (row['센터']) return row['센터'].trim();
  // 2순위: 이름 컬럼이 "(R_대전)이은정" 형식이면 파싱
  const m = (row['이름'] || '').match(/\(R_(.+?)\)/);
  return m ? m[1] : null;
}
```

### 3-2. OB담당자 → 복합키 변환

```js
function parseObName(obName) {
  // "(R_대전)이은정" → { center: 'daejeon', name: '이은정', key: '대전_이은정' }
  const m = obName.match(/\(R_(.+?)\)(.+)/);
  if (!m) return null;
  return {
    center: m[1] === '대전' ? 'daejeon' : m[1] === '유성' ? 'yuseong' : null,
    name: m[2].trim(),
    key: `${m[1]}_${m[2].trim()}`,
  };
}
```

### 4-1. 공휴일 판정 (★ 임시공휴일 대응)

```js
// 1단계: 기본 공휴일 테이블 (2040년까지 하드코딩 + 자동 계산)
const FIXED_HOLIDAYS = [
  // 매년 고정 공휴일
  '01-01',        // 신정
  '03-01',        // 삼일절
  '05-05',        // 어린이날
  '06-06',        // 현충일
  '08-15',        // 광복절
  '10-03',        // 개천절
  '10-09',        // 한글날
  '12-25',        // 성탄절
];

// 음력 공휴일 (설/추석/석가탄신일) 은 2040년까지 계산된 배열로 하드코딩
// (음력 계산 로직 포함하거나 정적 테이블 관리)
const LUNAR_HOLIDAYS = {
  '2026': ['2026-02-16', '2026-02-17', '2026-02-18',  // 설
           '2026-05-24',                                 // 석가탄신일
           '2026-09-25', '2026-09-26', '2026-09-27'],  // 추석
  '2027': [...],
  // ... 2040년까지
};

// 2단계: 임시공휴일 추가 테이블 (localStorage + UI로 관리)
// 키: 'grade_extra_holidays'
// 값: ['2026-05-04', '2026-08-17', ...]
function getExtraHolidays() {
  return JSON.parse(localStorage.getItem('grade_extra_holidays') || '[]');
}

function isHoliday(d) {
  if (d.getDay() === 0 || d.getDay() === 6) return true;  // 주말
  const mmdd = fmt(d, 'MM-DD');
  if (FIXED_HOLIDAYS.includes(mmdd)) return true;
  const yyyymmdd = fmt(d, 'YYYY-MM-DD');
  const year = d.getFullYear();
  if ((LUNAR_HOLIDAYS[year] || []).includes(yyyymmdd)) return true;
  if (getExtraHolidays().includes(yyyymmdd)) return true;  // 임시공휴일
  return false;
}
```

## 6. 집계 기간 정의

### 6-1. 회차 (3일 단위)

- 1 회차 = **3일** (영업일 기준)
- 회차 경계: db 파일의 `분배일 최대값` 기준 3일 전 ~ 당일
- 예: `분배일 max = 2026-04-23` → 이번 회차 = `2026-04-21 ~ 2026-04-23`

### 6-2. 집계 대상 기간/범위

| 항목 | 기간 | 범위 |
|------|------|------|
| DB수 | 이번 회차 (3일) | 모든 계열 |
| 계약수 | 이번 회차 (3일) | 모든 계열 |
| 취소수 **대상 계약** | **달력상 최근 3개월** | 모든 계열 |
| 취소 **발생 시점** | 이번 회차 (3일) | — |
| 근태 | 이번 회차 (3일, 주말·공휴일 제외) | 5가지 사례 |

### 6-3. "달력상 3개월" 정의

```js
function threeMonthsAgo(periodStart) {
  const d = new Date(periodStart);
  d.setMonth(d.getMonth() - 3);
  d.setDate(d.getDate() + 1);
  return d;
}

// 예시: 2026-04-21 → 2026-01-22
//       2026-03-31 → 2026-01-01
//       2026-05-01 → 2026-02-02
```

> ⚠️ 월말 경계 주의: `setMonth`는 자동 보정(Feb 30 → Mar 2)하므로 테스트 케이스 필수.

## 7. KPI 집계 공식

### 7-1. DB수 (`countDb`)

```
DB수 = count(
  분배일 ∈ [period.start, period.end] AND
  공통필터 ①②③④⑤⑥ 통과
)  — 복합키별
```

### 7-2. 계약수 (`countContracts`) ★ 당일 취소 제외한 실계약수

```
계약수 = count(
  오더일 ∈ [period.start, period.end] AND
  공통필터 ①②④⑤⑥ 통과 AND   ※ 재콜 ③ 미적용
  classifyStatus == '계약'     ※ '취소', '제외', '기타' 모두 불포함
)  — 복합키별
```

> 📌 **H 탈출 기준인 "계약 2건"은 이 `contract_count`(실계약수) 기준**이다. 취소 차감 전 값.

### 7-3. 취소수 (`countCancels`) ★ 별도 파일 + 고객키 매칭

```
취소수 = count(
  cancel.반품일 ∈ [period.start, period.end] AND
  cancel.고객키 ∈ cont.customerKey           ← 고객키 매칭
  AND 매칭된 cont.오더일 ∈ [threeMonthsAgo(period.start), period.end]
  AND 매칭된 cont.obName 공통필터 ①②⑥ 통과   ※ 재콜 ③ · DB정보 ④ · 주말 ⑤ 미적용
)  — 매칭된 담당자 복합키별
```

**처리 절차**
1. cancel 파일에서 반품일이 이번 회차 안에 있는 행 추출
2. 각 행의 고객키로 cont 에서 **해당 고객의 가장 최근 계약** 찾기
   - 동일 고객키의 계약이 여러 개면 **오더일 가장 최근 것**
3. 매칭된 계약의 오더일이 달력상 3개월 내인지 확인
4. 매칭된 계약의 담당자 복합키에 취소수 +1

### 7-4. 취소 반영 계약수 / 계약률 ★ 등급 배정 핵심 지표

```
reflected_contract = contract_count - cancel_count     // 음수 가능
reflected_rate     = reflected_contract / db_count     // db=0이면 0, 음수 가능
raw_rate           = contract_count / db_count          // 참고용 (취소 미반영)
```

> ⚠️ `reflected_contract`·`reflected_rate` 모두 음수 가능. UI에서 빨간색 볼드.
>
> ⚠️ **계약수가 취소 미포함** 이므로 `reflected_contract = contract_count - cancel_count` 공식은 **이중 차감이 발생하지 않음** (N그룹의 `실계약수`와 다른 점).

### 7-5. 총계약수 (`total_contract`) — 프로모션용

```
total_contract = contract_count   // 옵션 A 확정: 이번 회차 실계약수와 동일
```

프로모션 규칙 "B/C 총계약수 1위"의 기준값. 별도 컬럼으로 두지 않고 `contract_count`를 그대로 사용.

## 8. 등급 자동 배정

### 8-1. 경쟁 풀 — **대전·유성 통합 풀** ★

대전과 유성을 하나의 풀로 경쟁. 센터별 분리 경쟁 아님.

```js
gradePool = [
  ...competitive_from_daejeon,
  ...competitive_from_yuseong
];
```

### 8-2. 기본 규칙 (`assignGroupsSmart`)

1. **인원 분류**:
   - `is_resigned` → 퇴사 (제외)
   - `is_newbie` → 신입 (판정 별도, §8-9)
   - `is_confirmed_h` → 확정 H, 별도 H 트랙 (§8-3)
   - 나머지 → `gradePool`

2. `gradePool` 을 **`reflected_rate` 내림차순** 정렬 (동률 시 §7 순차 적용)

3. **비중 컷라인 (참고값, 절대값 아님)**:
   ```
   n = len(gradePool)
   A_count = max(1, Math.round(n * 0.25))
   B_count = max(1, Math.round(n * 0.40))
   C_count = n - A_count - B_count
   ```

4. 위에서부터 A → B → C 배정

> 📌 **비중은 참고값**. 프로모션·근태강등·신입보호 등 이후 로직으로 최종 비율이 25/40/35와 달라질 수 있으며, 이는 정상 동작이다.

### 8-3. 동점자 보정

같은 `reflected_rate`(소수 4자리 반올림)는 **절대 다른 등급으로 나뉘지 않는다**.
- A/B 경계의 `cut1` 값이 `cut1-1`과 같으면 → `cut1++` (위 등급으로 올림)
- B/C 경계도 동일
- 조정 후에도 동률이면 반복 (무한 루프 방지 상한 필요)

### 8-4. H 탈출 ★ streak 리셋

```
if (previous_grades[0] == 'H') AND (stats[key].contract_count >= 2):
    → 'C'
    → escaped_from_h[key] = true       // 프로모션 제외용
    → streak[key] = 0                  // ★ streak 완전 리셋
```

- **계약 2건 = `contract_count` (실계약수, 취소 반영 전)**
- H 탈출 후 다시 C가 반복되면 **streak는 0부터 새로 카운트**

**반복 예시**:
```
C(1) → C(2) → C(3) → C(4) → H → [탈출, streak=0] → C(1) → C(2) → C(3) → C(4) → H
```

### 8-5. H 진입

```
streak = 0
for g in previous_grades (최신 → 과거 순):
    if g == 'C': streak += 1
    else: break   // H가 섞이면 리셋됨 (§8-4 에서 이미 0됐을 것)

if (이번 회차 등급 == 'C') AND (streak >= 4):
    이번 회차 등급 = 'H'
    streak = 0
```

- **streak 카운팅 시작점**: `STREAK_START = '260421'` (정책 시행일). 이전 이력은 무시.
- IDB `results` 스토어의 YYMMDD 오름차순을 역순 탐색.

### 8-6. DB 0건 처리

```
if (db_count == 0) AND (NOT legitimate_absence):
    // 랭크 최하단으로 강제 이동, 결과적으로 C 배정
    sortKey = -Infinity
    // 경쟁 풀 정렬 시 제일 아래
    // 운영자가 이상 감지 시 수동 오버라이드 가능
```

### 8-7. 정당 사유 미출근 ★ 3일 **전부** 미출근일 때만

```
// 3일 중 하루라도 출근했으면 정상 성과 산정
// 3일 모두 미출근(휴가/연차/공가/경조)일 때만 "정당 사유 회차"

if (person.legitimate_absence == true) {
  grade = 가장 최근 출근 회차의 등급;   // 여러 회차 연속 정당사유면 그 이전 출근 회차까지 거슬러 올라감
  streak 카운팅에서 이 회차는 제외;    // H 진입 streak 계산 시 skip
  return;  // 프로모션·근태강등 모두 skip
}

// 랭크 상의 위치: 해당 등급 내 최하단
```

**예시**
- 직전 회차: B (출근)
- 이번 회차: 정당 사유 (3일 모두 미출근) → B 유지
- 다음 회차: 정당 사유 (계속 미출근) → 여전히 B 유지 (직전 출근 회차 기준)
- 다음다음 회차: 출근 → 정상 성과 산정

### 8-8. 프로모션 (B/C 총계약수 1위)

```js
// B → A
B_pool = [p for p in grades if grades[p]=='B' 
         AND !protectedMembers.has(p)          // 신입 보호 제외
         AND !legitimate_absence[p]];          // 정당사유 제외
if (B_pool.length) {
  top = argmax(total_contract).tiebreak(reflected_rate)(B_pool);
  grades[top] = 'A';
}

// C → B
C_pool = [p for p in grades if grades[p]=='C' 
         AND !escaped_from_h[p]                 // H 탈출자 제외
         AND !protectedMembers.has(p)          // 신입 보호 제외
         AND !legitimate_absence[p]];          // 정당사유 제외
if (C_pool.length) {
  top = argmax(total_contract).tiebreak(reflected_rate)(C_pool);
  grades[top] = 'B';
}
// 1위가 제외 대상이면 → 차순위에게 적용
```

### 8-9. 근태 강제 강등 ★ 하나씩 강등, C는 C 유지

근태이슈 2회 이상이면 **한 단계 강등**:
- A → B
- B → C
- **C → C (유지, 더 떨어지지 않음)**
- H → H (해당 없음, 이미 최하위)

```js
if (person.attendance_issues >= 2) {
  if (grades[key] == 'A') grades[key] = 'B';
  else if (grades[key] == 'B') grades[key] = 'C';
  // C, H 는 유지
}
```

### 8-10. 신입 보호 ★ 입사 +60일 고정

```js
const HIRE_PROTECT_DAYS = 60;

function isNewbieProtected(hireDate, now) {
  const protectEnd = new Date(hireDate);
  protectEnd.setDate(protectEnd.getDate() + HIRE_PROTECT_DAYS);
  protectEnd.setHours(23, 59, 59, 999);
  return now <= protectEnd;
}

if (isNewbieProtected(hireDates[key], now)) {
  // 신입은 최저 B, 최고 A
  // 성적에 따라 A는 가능 (하락 방지만 함)
  if (grades[key] == 'C' || grades[key] == 'H') {
    grades[key] = 'B';
    protectedMembers.add(key);
  }
  // history 에도 보호된 등급(B)으로 기록됨
  // → 보호 해제 후 이력 참조 시 C가 아닌 B로 보임
}
```

- 신입이 실적 좋으면 **A로 배정 가능** (상향은 제한 없음, 하락만 보호)
- 보호 대상자는 표·엑셀에서 **B 등급 최하단**에 정렬
- history 에는 보호된 등급(B)으로 저장 → H 진입 streak 계산에 영향

### 8-11. streak_adj 조정치 (관리자 수동 조정)

관리자가 특정 인원의 streak 값을 강제로 조정할 수 있는 기능.

```js
// localStorage 'grade_streak_adj' = { key: ±N, ... }
const adj = JSON.parse(localStorage.getItem('grade_streak_adj') || '{}');

function getStreak(key) {
  const rawStreak = computeStreakFromHistory(key);
  return Math.max(0, rawStreak + (adj[key] || 0));
}
```

**UI**: 관리자 메뉴에 "streak 조정" → 코디 선택 → ±값 입력.
**사용 예**: 이력 오류 보정, 특수 상황(질병 휴직 후 복귀) 등.

### 8-12. 수동 오버라이드 (최종 덮어쓰기)

관리자가 자동 산정 결과를 임의로 변경.

```js
// IDB results[periodId].rows[key].manualGrade = 'A' 같은 형태로 저장

// 파이프라인 마지막 단계에서:

  grades[key] = overrides[key];
  // 이 때는 이력에도 수동 값이 저장됨
```

## 9. 출력 형식 — 엑셀 시트

### 9-1. 시트 1개 = 회차 1개

### 9-2. 시트 상단
- D2:K3 병합: "뇌새김영어 대전·유성 3일 차등 성과"

### 9-3. 데이터 컬럼

```
등급 | 센터 | 코디명 | 직전등급 | DB수 | 계약수 | 취소수 |
취소반영 계약수 | 취소반영 계약률 | 취소미반영 계약률 | 총계약수 |
```

- 기본 정렬: `reflected_rate` DESC

### 9-4. 행 정렬
1. 등급 순: A → B → C → H
2. 등급 내: `reflected_rate` DESC
3. 신입 보호자는 B 등급 **최하단**
4. 정당 사유 인원은 해당 등급 **최하단**
5. DB 0건 인원은 C 등급 **최하단**
6. 합계행: 첫 데이터 행 위쪽

```js
  const backup = snapshotAll();  // 현재 상태 스냅샷
  try {
    await IDB.transaction(['results', 'history'], 'readwrite', async (tx) => {
      await tx.objectStore('results').put(data, periodId);
      await updateHistory(data);
    });
    downloadBackup(periodId);  // 성공 후 백업 파일 생성
  } catch (err) {
    restoreFromSnapshot(backup);  // 실패 시 롤백
    alert('저장 실패. 이전 상태로 복구되었습니다: ' + err.message);
  }
}
```

## 16. 용어 정리

| 용어 | 의미 |
|------|------|
| 회차 | 3일 단위 집계 기간 |
| 복합키 | `센터_이름` (동명이인 처리) |
| DB수 | 이번 회차 분배받은 유효 DB 건수 (모든 계열) |
| 계약수 | 이번 회차 `고객상태 == 계약` 건수 (당일 취소 미포함) |
| 취소수 | 반품 파일의 반품일 기준, 고객키 매칭 + 오더일 달력상 3개월 내 |
| 취소 반영 계약수 | 계약수 - 취소수 (음수 가능) |
| 취소 반영 계약률 | 취소 반영 계약수 / DB수 (음수 가능) |
| 경쟁 대상 (gradePool) | 대전·유성 통합 경쟁 풀 (퇴사·신입·확정H 제외) |
| 확정 H | 지난 회차 H, 계약 2건 달성 시 C로 탈출 (streak 리셋) |
| streak | 연속 C 회차 수 (H 탈출 시 0 리셋) |
| streak_adj | 관리자 수동 streak 조정치 |
| 정당 사유 미출근 | 3일 **전체** 휴가/연차/공가/경조 → 직전 출근 등급 유지 |
| 근태이슈 | 지각 / 오전결근 / 오후결근 / 결근 / 조퇴 (CMS 2-8 기준) |
| 신입 보호 | 입사 +60일 이내, C/H 배정 시 B 상향 (A는 허용) |
| 수동 오버라이드 | 관리자가 자동 계산 결과를 임의로 변경 |
| 달력상 3개월 | `setMonth(-3) + 1일` (90일 아님) |
