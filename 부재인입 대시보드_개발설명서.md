# 부재 인입 분석 대시보드 개발 명세서 (v1.1)

> **개발 환경:** Visual Studio Code + Claude AI
> **실행 방식:** 브라우저 단독 실행형 SPA (서버 불필요)
> **데이터 보관:** 브라우저 IndexedDB (로컬, 누적 저장)
> **버전 이력:** v1.0 (초안) → v1.1 (DB/계약 컬럼 구조 확정, 법인별 분리 업로드 반영)

---

## 1. 프로젝트 개요

### 1.1 목적
고객이 부재 후 콜백한 건들 중 **당사가 미수신한 부재 인입 건수와 미수신률**을 일자별/주차별/월별로 추적하기 위한 대시보드를 구축한다.

### 1.2 주요 산출물
- 부재 인입 건수 (일/주/월)
- 미수신률 (%)
- 법인별(위버스마인드 / 더블유케어) 분리 집계
- 상세 부재 인입 케이스 리스트

### 1.3 핵심 제약 조건
- IN call 데이터와 콜백 데이터의 파일 크기가 매우 큼 → **서버 저장 없이 브라우저에서 처리**
- 데이터는 **누적 업로드** 방식 (덮어쓰기 X, 신규 데이터만 append)
- 중복 업로드 방지 로직 필수

## 3. 데이터 소스 정의

### 3.1 IN call 데이터 (인입 콜 로그) - 법인 공통

| 컬럼 | 명칭 | 설명 |
|------|------|------|
| A | 번호 | 일련번호 |
| B | 아이디 | 콜 받은 상담원 아이디 |
| C | 상담원명 | - |
| D | 내선번호 | - |
| E | 일자 | 전화 들어온 일자 |
| F | 통화시작 | 통화 시작 시간 |
| G | 통화종료 | 통화 종료 시간 |
| H | 통화시간 | 초 단위 |
| I | 그룹 | 법인 구분용 키워드 포함 |
| J | 대표번호 | **앞자리 0이 빠진 형태** (예: `25210868`) |
| K | 발신자번호 | `0##-####-####` 형식 |
| L | 상태 | In 연결, 호분배 연결, 호분배 실패, 호분배 재연결 등 |

**파싱 시 주의사항:**
- 시트가 10,000행 단위로 분할되어 있음 → **모든 시트를 순회**해야 함
- 각 시트의 **1~2행은 병합 가비지 데이터**, **3행이 컬럼명**, **4행부터 실데이터**
- 시트별 컬럼 순서가 동일하다고 가정하지 말고, 3행 헤더를 매번 매핑

**법인 구분 규칙 (I열 그룹 기반):**
```
I열에 "뇌새김" 포함 → 위버스마인드
I열에 "케어" 포함 → 더블유케어
그 외 → 미분류 (집계 제외, 단 경고 표시)
```

**유효 통화 판정:**
```
조건1: L열(상태)에 "연결" 포함
조건2: H열(통화시간) >= 90초
조건1 AND 조건2 → 유효 통화
```

### 3.2 콜백 데이터 (전체 인입 모수) - 법인 공통

| 컬럼 | 명칭 | 설명 |
|------|------|------|
| A | 대표번호 | **0 포함** (예: `025210868`) |
| B | 발신자번호 | `0##########` 형식 |
| C | 일별 중복 총횟수 | 같은 고객 일일 반복 콜 수 |
| D | 녹음파일수 | - |
| E | call시간 | 콜 들어온 시간 |

**용도:** 전체 인입 콜의 모수. 부재 인입 판정의 출발점.

**법인 구분 방법:** 콜백 데이터 자체에는 법인 정보가 없음 → **대표번호 매핑 테이블(Section 3.5)을 통해 매칭**

**용도:** 콜백한 발신자번호가 우리 DB에 있는 고객인지 확인 (계약 데이터에 없는 경우의 2차 체크)

### 3.4 계약 데이터 (법인별 분리 업로드)

위버스마인드용 1개, 더블유케어용 1개 → **총 2개 파일을 각각 별도 업로드.**
**컬럼 구조는 DB 데이터와 100% 동일** (Section 3.3 표 그대로).

**핵심 컬럼:**
- **M, N, O** — 발신자번호 매칭 대상 (3개 OR)
- **X (오더일)** — `yyyy-mm-dd hh:mm:ss` 형식, 계약 시점 비교 기준

**유효 계약 판정:**
- **X열(오더일)이 값이 있는 행만 유효 계약으로 인정**
- X열이 비어있는 행은 무시

---

### 3.5 대표번호 ↔ 법인 매핑 테이블 (1회 등록 후 유지)

**별도 등록 필요** — 콜백 데이터에는 법인 정보가 없기 때문에 필수.

| 컬럼 | 명칭 | 예시 |
|------|------|------|
| A | 대표번호 (0 포함 원본) | `0212345678` |
| B | 법인 | `위버스마인드` 또는 `더블유케어` |

**번호 정규화 규칙:**
- IN call J열 = 앞자리 0 없음
- 콜백 A열 = 앞자리 0 있음
- 매핑 테이블 A열 = 앞자리 0 있음
- → **모든 비교는 "앞자리 0 제거 + 숫자만 남기기" 정규화 후 수행**

```typescript
function normalizePhone(phone: string | number): string {
  return String(phone).replace(/[^0-9]/g, '').replace(/^0+/, '');
}
```

## 4. 부재 인입 판정 로직 (핵심 알고리즘)

### 4.1 처리 순서 (콜백 데이터 1건당)

```
[Step 1] 콜백 데이터 1건 추출 → (대표번호_norm, 발신자번호_norm, call시간)
   ↓
[Step 2] 대표번호 매핑 테이블 조회 → 법인 결정
   - 매핑 없음 → "미분류"로 별도 카운트, 본 집계 제외
   ↓
[Step 3] 해당 법인의 계약 데이터에서 발신자번호 조회
   매칭 조건: 발신자번호_norm이 (M_norm OR N_norm OR O_norm) 중 하나와 일치
   - 매칭된 계약 중 "X열(오더일) <= 콜백 call시간"인 행이 1건이라도 있음 → 제외 (계약 후 콜백)
   - 매칭된 계약 모두 "X열(오더일) > 콜백 call시간" → Step 5로 (콜백이 계약 이전)
   - 매칭된 계약 없음 → Step 4로
   ↓
[Step 4] 해당 법인의 DB 데이터에서 발신자번호 조회
   매칭 조건: 발신자번호_norm이 (M_norm OR N_norm OR O_norm) 중 하나와 일치
   - 매칭 없음 → 제외 (모수 아님)
   - 매칭 있음 → Step 5로
   ↓
[Step 5] IN call 데이터에서 매칭 레코드 검색
   매칭 조건 (모두 충족):
     ① 발신자번호 일치 (K열 정규화 = 콜백 B열 정규화)
     ② 대표번호 일치 (J열 정규화 = 콜백 A열 정규화)
     ③ |IN call 통화시작(F) - 콜백 call시간(E)| <= 2분 (절대값)
   ↓
[Step 6] 부재 인입 판정
   - IN call 매칭 0건 → 부재 인입 (시스템 누락 / 호분배 실패 미기록)
   - IN call 매칭 있음 + 모든 매칭 레코드가 유효통화 아님 → 부재 인입
   - IN call 매칭 있음 + 1건이라도 유효통화 → 정상 (제외)
```

### 4.2 확정된 결정 사항

| 항목 | 결정 |
|------|------|
| IN call에 매칭 없는 콜백 | **부재 인입으로 카운트** |
| 집계 단위 | **콜 단위 + 고객 단위 둘 다 제공** |
| 시간 비교 방향 | **절대값 ±2분** |
| 계약 후 제외 시점 | **콜백 시점보다 이전 계약(X열 오더일)이 1건이라도 있으면 제외** |
| 발신자번호 매칭 범위 | **M / N / O열 3개 OR 매칭** |
| 유효 계약 판정 | **X열(오더일) 값이 있는 행만 유효 계약** |
| 데이터 보관 | **브라우저 IndexedDB 누적** |

## 5. 미수신률 계산

### 5.1 정의

```
미수신률(%) = (부재 인입 콜 수) / (분모) × 100
```

### 5.2 분모 옵션 (둘 다 화면에 표시)

| 지표명 | 분모 정의 | 의미 |
|--------|-----------|------|
| 전체 미수신률 | 전체 콜백 콜 수 (법인 매핑된 것) | 전체 인입 대비 부재 비율 |
| 유효 미수신률 ★ | 계약(콜백 이전) 또는 DB 보유 고객의 콜백 콜 수 | **주력 지표** - 유효 모수 기준 부재 비율 |

## 7. IndexedDB 스키마 설계

### 7.1 테이블 구성 (Dexie.js)

```typescript
db.version(1).stores({
  // IN call 로그
  in_calls: '++id, &uniqueKey, 발신자번호_norm, 대표번호_norm, 통화시작_ts, 법인, 유효통화여부',

  // 콜백 (전체 인입)
  callbacks: '++id, &uniqueKey, 발신자번호_norm, 대표번호_norm, call시간_ts, 법인',

  // DB 데이터 (법인 통합 저장, 행별 법인 태깅)
  customer_db: '++id, &uniqueKey, 법인, [법인+M_norm], [법인+N_norm], [법인+O_norm]',

  // 계약 데이터 (법인 통합 저장, 행별 법인 태깅)
  contracts: '++id, &uniqueKey, 법인, 오더일_ts, [법인+M_norm], [법인+N_norm], [법인+O_norm]',

  // 대표번호 ↔ 법인 매핑
  number_mapping: '&대표번호_norm, 법인',

  // 업로드 이력
  upload_history: '++id, fileHash, fileName, dataType, 법인, uploadedAt, rowCount',

  // 분석 결과 캐시
  analysis_cache: '&cacheKey, data, computedAt'
});
```

### 7.2 uniqueKey 생성 규칙 (중복 방지)

```typescript
// IN call
uniqueKey = `${발신자번호_norm}_${대표번호_norm}_${통화시작_ts}_${통화시간}`

// 콜백
uniqueKey = `${발신자번호_norm}_${대표번호_norm}_${call시간_ts}`

// DB / 계약
uniqueKey = `${법인}_${F고객번호 || G고객키}_${X오더일_ts || I분배일_ts || A번호}`
// → 식별자가 일관되지 않을 경우를 대비해 fallback 다단계
```

### 7.3 인덱스 전략

- **복합 인덱스** `[법인+M_norm]`, `[법인+N_norm]`, `[법인+O_norm]` → 발신자번호 매칭 시 법인별 빠른 조회
- 날짜는 모두 **timestamp (number)** 로 저장 → range query 효율
- 분석 결과는 **(필터 조합 해시)** 를 cacheKey로 저장 → 동일 필터 재계산 방지

## 8. 데이터 처리 파이프라인

### 8.1 업로드 단계

```
[파일 드롭 + 법인 선택(DB/계약만)]
   ↓
[파일 해시 계산 → 중복 업로드 체크]
   ↓
[SheetJS로 파싱]
   - IN call: 모든 시트 순회, 3행 헤더, 4행부터
   - 콜백/DB/계약/매핑: 1행 헤더, 2행부터
   ↓
[정규화]
   - 전화번호: 0 제거 + 숫자만
   - 날짜: timestamp 변환
   ↓
[법인 분류]
   - IN call: I열(그룹) 키워드로 자동
   - 콜백: 매핑 테이블 lookup
   - DB/계약: 업로드 시 사용자 선택값 사용
   ↓
[유효통화 판정 (IN call만)]
[유효계약 판정 (계약만 - X열 값 유무)]
   ↓
[uniqueKey 생성 → IndexedDB bulkPut]
   ↓
[업로드 이력 기록]
[분석 캐시 무효화]
```

### 8.2 분석 단계 (Web Worker)

```
[필터 적용: 기간/법인/단위]
   ↓
[캐시 확인 → hit 시 즉시 반환]
   ↓
[콜백 데이터 조회 (기간 + 법인 필터)]
   ↓
[각 콜백에 대해 Section 4.1 판정 로직 실행]
   - 계약 lookup (법인별 인덱스 활용, M/N/O OR)
   - DB lookup (법인별 인덱스 활용, M/N/O OR)
   - IN call lookup (발신자번호 + 대표번호 + ±2분)
   ↓
[부재 인입 케이스 수집 + 사유 태깅]
   ↓
[일/주/월 단위 그룹핑]
   ↓
[KPI 계산 (콜/고객 단위 × 전체/유효 분모)]
   ↓
[캐시 저장 → 결과 반환]
```

### 8.3 성능 최적화 포인트

- 대용량 분석은 **Web Worker**에서 실행 (UI 블로킹 방지)
- 발신자번호 매칭은 **(법인, 번호) 복합 인덱스**로 인메모리 해시맵 사전 구축
- 콜백 1건씩 순회하지 말고, 발신자번호별로 그룹핑 후 일괄 매칭
- IN call의 ±2분 비교는 발신자번호로 1차 필터 → 시간 범위로 2차 필터

## 12. 엣지 케이스 처리

| 케이스 | 처리 방법 |
|--------|-----------|
| 대표번호 매핑에 없는 콜백 | "미분류"로 별도 카운트, 메인 집계 제외, 경고 배지 표시 |
| IN call 그룹에 "뇌새김"도 "케어"도 없음 | "미분류" 분류, 집계 제외 |
| 같은 고객이 같은 날 여러 번 부재 인입 | 콜 단위 = N, 고객 단위 = 1 (둘 다 표시) |
| 콜백 시간이 미래 시점 (데이터 오류) | 경고 로그 + 분석 제외 |
| 통화시간 음수/비정상 | 0 처리, 유효통화 판정 자동 fail |
| 계약 데이터에 동일 고객 다건 | 콜백 이전 X열(오더일) 1건이라도 있으면 제외 |
| 시트 헤더가 예상과 다름 | 파싱 실패 알림 + 시트명·행번호 표시 |
| 계약 X열(오더일) 빈 값 | 무효 계약으로 처리, 모수에서 제외 |
| M/N/O 중 일부만 채워진 경우 | 빈 값은 매칭 후보에서 자동 제외 (정규화 후 빈 문자열) |
| 같은 발신자번호가 위버스/W케어 양쪽 DB에 있음 | **법인별로 독립 판정** (콜백의 대표번호가 결정한 법인만 사용) |
| 매핑 테이블 미등록 상태에서 분석 시도 | 안내 모달 + 매핑 업로드 유도 |
